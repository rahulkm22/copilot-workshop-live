# Task Manager Technical Design

## 1. Data models

### Task

| Property | Type | Required | Validation rules |
|---|---|---|---|
| `id` | string | Yes | Required on persisted objects; must be system-generated (not user-supplied); must match `^task-[0-9]{13}$`; must be unique in store; immutable after creation. |
| `title` | string | Yes | Required; must be a string; trim leading/trailing whitespace before validation; must be 1-100 characters after trim; reject empty/whitespace-only values. |
| `description` | string | No | Optional; if omitted default to empty string; if provided must be a string; trim before persistence; max length 500 characters after trim. |
| `status` | `'todo' \| 'in-progress' \| 'done'` | Yes | Required on persisted objects; default to `'todo'` during create; allowed values only: `todo`, `in-progress`, `done`; accept case-insensitive input and normalize to lowercase canonical value. |
| `priority` | `'low' \| 'medium' \| 'high'` | Yes | Required on persisted objects; default to `'medium'` during create; allowed values only: `low`, `medium`, `high`; accept case-insensitive input and normalize to lowercase canonical value. |
| `createdAt` | number | Yes | Required; must be an integer Unix timestamp in milliseconds (`Number.isInteger`); must be generated by system at creation; must be greater than 0; immutable after creation. |
| `updatedAt` | number | Yes | Required; must be an integer Unix timestamp in milliseconds (`Number.isInteger`); set at creation and on each mutation; must be greater than or equal to `createdAt`; never decreases between updates. |
| `category` | string | No | Optional; if omitted default to `'general'`; must be a string; trim leading/trailing whitespace before validation; must be 1-50 characters after trim; reject empty/whitespace-only values; accept case-insensitive input and normalize to lowercase canonical value. |

### Command input model

| Input | Type | Required | Validation rules |
|---|---|---|---|
| `command` | string | Yes | Must be one of `add`, `list`, `update`, `done`, `delete`, `help`. |
| `id` | string | Conditional | Required for `update`, `done`, `delete`; must map to an existing task id. |
| `--title` | string | Conditional | For `add`: required. For `update`: optional. Same validation as `Task.title`. |
| `--description` | string | No | Optional for `add` and `update`; same validation as `Task.description`. |
| `--status` | string | No | Must be `todo`, `in-progress`, or `done` (case-insensitive). |
| `--priority` | string | No | Must be `low`, `medium`, or `high` (case-insensitive). |
| `--category` | string | No | Optional for `add` (defaults to "general"). For `list`: optional filter to display only tasks matching the specified category (case-insensitive). |
| `--sort` | string | No | Must be `date` or `priority`. |

## 3. Module responsibilities

### `src/index.js`
- Exports: none (executable entrypoint).
- Responsibility: parse raw CLI args, dispatch command handlers, set exit codes.
- Depends on: `src/utils/parser.js`, `src/commands/*.js`, `src/store/task-store.js`, `src/utils/formatter.js`.

### `src/commands/add.js`
- Exports: `addCommand(args, store)`.
- Responsibility: validate add input, create task data, persist in store, return display payload.
- Depends on: `src/utils/validators.js`, `src/models/task.js`, `src/store/task-store.js`.

### `src/commands/list.js`
- Exports: `listCommand(args, store)`.
- Responsibility: validate list filters/sort, query store, return formatted list payload.
- Depends on: `src/utils/validators.js`, `src/store/task-store.js`.

### `src/commands/update.js`
- Exports: `updateCommand(args, store)`.
- Responsibility: ensure target id exists, validate at least one mutable field, apply patch, refresh `updatedAt`.
- Depends on: `src/utils/validators.js`, `src/store/task-store.js`.

### `src/commands/done.js`
- Exports: `doneCommand(args, store)`.
- Responsibility: set `status` to `done` for an existing task and update timestamp.
- Depends on: `src/store/task-store.js`.

### `src/commands/delete.js`
- Exports: `deleteCommand(args, store)`.
- Responsibility: delete task by id and return deletion confirmation payload.
- Depends on: `src/store/task-store.js`.

### `src/commands/help.js`
- Exports: `helpCommand()`.
- Responsibility: return static usage/help text.
- Depends on: none.

### `src/models/task.js`
- Exports: `createTask(input, now)`, `normalizeTaskPatch(input)`.
- Responsibility: construct canonical Task objects and normalize model fields.
- Depends on: `src/utils/validators.js`.

### `src/store/task-store.js`
- Exports: `TaskStore` class.
- Responsibility: hold in-memory task collection and implement add/get/list/update/delete/query operations.
- Depends on: none.

### `src/utils/parser.js`
- Exports: `parseArgs(argv)`.
- Responsibility: convert argv into `{ command, positional, flags }` shape.
- Depends on: none.

### `src/utils/formatter.js`
- Exports: `formatTask(task)`, `formatTaskList(tasks)`, `formatError(message)`.
- Responsibility: keep user-facing output format consistent.
- Depends on: none.

### `src/utils/validators.js`
- Exports: `validateTitle`, `validateDescription`, `validateStatus`, `validatePriority`, `validateSort`, `validateTaskId`.
- Responsibility: centralize domain validation and normalization rules.
- Depends on: none.

## 4. Error handling strategy

### Error types

1. `ValidationError`
- Thrown when user input is syntactically valid CLI text but violates domain rules.
- Thrown from: `src/utils/validators.js`, surfaced by command modules.
- Example cases: empty title, invalid status, invalid sort option, description too long.

2. `NotFoundError`
- Thrown when referenced task id does not exist.
- Thrown from: `src/store/task-store.js` read/update/delete operations.
- Example cases: `task update <id>`, `task done <id>`, `task delete <id>` with unknown id.

3. `CommandError`
- Thrown when command structure is invalid or incomplete.
- Thrown from: `src/utils/parser.js` and command modules.
- Example cases: unknown command, missing required positional argument, update with no fields.

### Throwing and handling rules

- Command modules must validate inputs before mutating store state.
- Store methods throw `Error` subclasses with descriptive messages; never throw strings.
- `src/index.js` wraps dispatch in `try/catch`, writes failures with `console.error`, and exits with status code `1`.
- Successful command execution exits with status code `0`.
- Error message convention: `Error: <specific description>`.
